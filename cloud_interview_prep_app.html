Cloud fundamentals
Core concepts every cloud engineer must know
Service models
IaaS  → you manage OS up        (VMs, networking)
PaaS  → you manage app only     (App Service, RDS)
SaaS  → you manage nothing      (Gmail, Office 365)

Rule: more control = more responsibility
CIDR notation — memorize this
/8   = 16 million IPs   (huge network)
/16  = 65,536 IPs       (typical VNet/VPC)
/24  = 256 IPs          (typical subnet)
/32  = 1 IP             (single host)

Smaller number = MORE IPs = BIGGER network
Uptime SLA numbers
99%     =  8.7 hours  downtime/year
99.9%   = 52.6 minutes downtime/year
99.95%  = 26  minutes  downtime/year
99.99%  =  5.2 minutes downtime/year  (HA target)
99.999% = 31  seconds  downtime/year  (five nines)
RTO vs RPO
RPO — Recovery Point Objective: how much data can you lose?
RTO — Recovery Time Objective: how long can the system be down?

Lower RPO/RTO = more expensive. Always define these before designing DR.
Zero Trust principles
Never trust, always verify — even internal traffic
Identity  → MFA + Managed Identity for services
Network   → private endpoints, WAF on public ingress
Data      → encryption at rest (CMK) + in transit (TLS 1.2+)
Access    → least privilege for every identity
Azure
Key services, networking, and architecture patterns
Load balancer — which one to use?
Global traffic?      → Azure Front Door + WAF
Regional L7 HTTP?    → Application Gateway v2
Regional L4 TCP/UDP? → Azure Load Balancer
DNS-based global?    → Traffic Manager
Managed Identity — no passwords!
Azure creates an identity for your App Service/VM automatically. It authenticates to Key Vault, SQL, Storage using this identity — zero passwords stored anywhere. Azure handles token issuance and rotation. This is the answer to "how do you handle secrets in Azure?"
Hub-Spoke network topology
Hub VNet: Firewall + Bastion + DNS (shared services)
  ├── Spoke 1: App A  (VNet Peering to Hub)
  ├── Spoke 2: App B  (VNet Peering to Hub)
  └── Spoke 3: App C  (VNet Peering to Hub)

All Spoke↔Spoke traffic routes via Hub Firewall

AWS
VPC design, key services, and architecture patterns
VPC subnet design
PUBLIC subnet  → has route to Internet Gateway
                 for: Load Balancers, NAT Gateway
PRIVATE subnet → no internet route
                 for: EC2, ECS (never expose app servers!)
DB subnet      → private, only app tier can reach
Security Groups vs NACLs
Security Group → stateful, instance level, allow only
NACL           → stateless, subnet level, allow + deny

Use SGs for most cases.
Use NACLs to block entire IP ranges at subnet level.
RDS Multi-AZ vs Read Replicas
Multi-AZ — synchronous standby, auto-failover ~60s, for HIGH AVAILABILITY. Standby cannot serve reads.

Read Replicas — async copy for READ PERFORMANCE. Up to 15 for Aurora. Can be cross-region.

Use BOTH together for HA + performance.
EC2 pricing models
On-Demand    → pay per hour, no commitment (expensive)
Reserved     → 1-3 year commit, save up to 72%
Spot         → unused capacity, save up to 90%
Savings Plan → flexible commitment (recommended)
GCP
Global VPC, BigQuery, GKE — GCP's strengths
GCP VPC is GLOBAL — key difference!
AWS/Azure VPC: regional (one per region)
GCP VPC:       global  (spans ALL regions!)

Subnets are regional. Resources in different
regions in the same VPC talk privately — no
extra peering needed!
Where GCP wins
Best Kubernetes — GKE (Google invented K8s) Best data warehouse — BigQuery (serverless) Global LB built-in (anycast, single IP) Cloud Spanner — global distributed SQL Cheapest preemptible VMs (91% off)
BigQuery in one paragraph
Serverless data warehouse — no cluster to manage. Petabyte queries in seconds. Pay per TB scanned (on-demand). Run ML models using SQL (BQML). Columnar storage. GCP's biggest differentiator. Use partitioning + clustering to reduce data scanned and cut costs.
Workload Identity in GKE
K8s pods authenticate to GCP services without service account JSON key files. K8s service account is linked to a GCP service account. Pods get short-lived tokens automatically. No secrets stored in pods or config files.

DevOps Engineer
CI/CD, IaC, containers, GitOps
CI/CD pipeline stages
1. git push → pipeline triggers
2. Build: compile + unit tests + Docker image
3. Scan: Trivy (container) + SAST + OWASP ZAP
4. Deploy staging → integration tests
5. Manual approval (optional gate)
6. Deploy prod → Blue/Green or Canary
7. Health check → auto-rollback if fail
Deployment strategies
Blue/Green — two environments, instant traffic switch, instant rollback. Double cost during cutover.

Canary — 5% → 20% → 50% → 100%. Monitor error rate at each step. Best for risky changes.

Rolling — update pods one by one. No extra infra. Both versions briefly co-exist.
Terraform workflow
terraform init     → download providers
terraform plan     → preview changes (review this!)
terraform apply    → create/update resources
terraform destroy  → delete all

State: store in Azure Blob / S3 / GCS
Locking: prevents two people applying at once
GitOps with ArgoCD
Git repo (K8s manifests)
  → ArgoCD watches for changes
  → detects drift between Git and cluster
  → auto-syncs cluster to match Git

No manual kubectl in prod. Changes via PRs.
Audit trail = Git history. Rollback = git revert.
Cloud Architect
HA design, migration, security, cost
HA design checklist
No single points of failure at every tier. Multi-AZ compute with auto-scaling. Multi-AZ database with auto-failover. Load balancer with health checks. Graceful degradation — serve cached data if DB is slow. Circuit breakers between services.
Migration — 6 Rs framework
Rehost     → lift-and-shift VMs as-is (fastest)
Replatform → swap in managed services (RDS, K8s)
Repurchase → move to SaaS (Salesforce, etc)
Refactor   → redesign as microservices/serverless
Retain     → keep on-prem (not ready to migrate)
Retire     → decommission (no longer needed)
Cost optimization levers
Right-size    → monitor CPU/RAM, downsize over-provisioned
Reserved      → 1-3 yr commit for baseline (save 40-70%)
Spot/Preempt  → for dev/test, batch jobs (save 60-90%)
Auto-scale    → scale down nights and weekends
Storage tiers → lifecycle policies for old data
Tag everything → cost allocation per team/product
DR strategies by cost
Backup/Restore  → cheapest, hours RTO
Pilot Light     → core services on, scale up on DR
Warm Standby    → scaled-down copy always running
Active/Active   → full capacity both regions, instant
Site Reliability Engineer
SLIs, SLOs, error budgets, incident response
SLI → SLO → SLA → Error Budget
SLI  = measurement   (99.95% requests succeed)
SLO  = target        (99.9% per month — internal)
SLA  = contract      (99.5% — always below SLO)
Error Budget = 100% - SLO = 0.1% = 43 min/month

High budget → move fast, ship features
Low budget  → slow down, fix reliability first
4 Golden Signals — monitor these always
1. Latency    → how long requests take (track p99!)
2. Traffic    → requests per second
3. Errors     → rate of failed requests (HTTP 5xx)
4. Saturation → CPU %, memory %, queue depth
Incident response flow
Detect → Acknowledge → Assess severity → Notify stakeholders → Mitigate first (restore service, root cause later) → Investigate → Post-mortem within 48hr (blameless, 5 Whys, action items with owners).
Toil — what it is and how to reduce it
Toil = manual, repetitive, automatable work with no lasting value. Examples: manually restarting services, manual scaling before events, responding to noisy alerts.

Reduce by: self-healing (liveness probes), auto-scaling, automated runbooks. Target: toil below 50% of SRE time.

Interview Q&A
Click any question to reveal the answer
What is the difference between NSG and Azure Firewall? ▼
NSG = simple allow/deny rules at subnet or NIC level, free, stateful. Azure Firewall = fully managed, FQDN filtering, IDPS, TLS inspection, centralized, costs money. Use BOTH: NSG for per-subnet micro-segmentation, Firewall for centralized east-west traffic control.
What is a Private Endpoint and why use it? ▼
A Private Endpoint gives a PaaS service (Azure SQL, Blob Storage) a private IP inside your VNet. Traffic never leaves Microsoft's network. Prevents data exfiltration. The service becomes accessible only from inside your VNet — no public internet path exists at all.
What is an SLO and how do you set one? ▼
SLO = Service Level Objective, your internal reliability target. To set one: look at historical data, understand what customers need, set SLO slightly below what you've historically achieved to give room. Example: historically 99.95% → set SLO at 99.9%. Review and adjust quarterly.
What is GitOps? ▼
GitOps = Git is the single source of truth for all infrastructure and app config. An operator (ArgoCD / Flux) continuously syncs the live cluster state to match what's in Git. Changes go via PRs — no manual kubectl in production. Audit trail in Git history, rollback via git revert.
RDS Multi-AZ vs Read Replicas — what's the difference? ▼
Multi-AZ = synchronous standby in another AZ, automatic failover in ~60 seconds, for HIGH AVAILABILITY. The standby cannot serve read traffic. Read Replicas = async copy for READ PERFORMANCE, up to 15 replicas for Aurora. Use both together for HA + read performance.
How do you do zero-downtime deployments? ▼
Blue/Green: two identical environments, instant traffic switch, instant rollback — double cost during cutover. Canary: route 5% to new version, monitor metrics, increase gradually. Rolling (K8s default): update pods one by one with readiness probes ensuring health before routing traffic.
What is Managed Identity in Azure? ▼
Azure creates an Azure AD identity for your resource (App Service, VM, AKS pod). The app authenticates to Key Vault, SQL, Storage using this identity — no passwords stored anywhere. Azure handles token issuance and automatic rotation. This eliminates stored credentials entirely.
How does GCP VPC differ from AWS VPC? ▼
GCP VPC is global — one VPC spans all regions, subnets are regional. AWS VPC is regional — separate VPCs per region, need Transit Gateway or peering for multi-region. GCP is simpler for global apps: resources in different regions in the same VPC communicate privately without any extra config.
What is Platform Engineering vs DevOps? ▼
DevOps engineer is embedded in a product team, builds CI/CD for that specific product. Platform Engineer builds internal tools and platforms that all engineering teams use. Platform team's customers are other developers. They create self-service infra, golden paths, and templates — reducing toil for every team simultaneously.
How do you handle secrets in Kubernetes? ▼
Never store raw secrets in Git. Use External Secrets Operator to sync from Key Vault / Secrets Manager / Vault into K8s Secrets automatically. Use Workload Identity / IRSA so pods authenticate without stored credentials. Rotate secrets in the central store — ESO syncs the rotation automatically.
